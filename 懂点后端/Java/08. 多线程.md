# 多线程

## 基本概念

| 维度 | 进程 | 线程 |
| --- | --- | --- |
| 定义 | 运行中的程序，是程序的一次执行过程，有产生、存在、消亡的生命周期 | 进程创建的一个实体，一个进程可拥有多个线程 |
| 资源分配 | 操作系统进行资源（内存等）分配的基本单位 | 共享所属进程的资源，不单独占有内存地址空间 |
| 调度单位 | 不直接参与 CPU 调度 | 操作系统进行 CPU 时间调度的基本单位 |

使用多线程而非多进程的原因：线程间通信比进程间简单、共享资源更容易，且线程是轻量级的，创建和切换的系统开销更小。

## 线程创建方式

**方式 1：继承 Thread 类**，重写 `run()`。

```Java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("通过继承 Thread 类创建线程");
    }
}
new MyThread().start();
```

调用 `start()` 后线程才真正启动，不可重复调用（否则抛异常）。缺陷：受 Java 单继承限制，且任务逻辑与线程机制耦合，仅适合快速演示。

**方式 2：实现 Runnable 接口**，将实例传给 `Thread` 构造器。

```Java
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("通过实现 Runnable 接口创建线程");
    }
}
new Thread(new MyRunnable()).start();
new Thread(() -> System.out.println("Lambda Runnable")).start(); // Lambda 写法
```

避免单继承限制，任务与线程解耦，同一 Runnable 可被多个线程共享，是最常用的方式。缺陷：`run()` 返回 void，无法获取执行结果或抛受检异常。

**方式 3：实现 Callable 接口 + FutureTask**，可返回结果并抛异常。

```Java
class MyCallable implements Callable<String> {
    @Override
    public String call() throws Exception {
        Thread.sleep(1000);
        return "Callable 执行完毕，返回结果";
    }
}
FutureTask<String> futureTask = new FutureTask<>(new MyCallable());
new Thread(futureTask).start();
String result = futureTask.get(); // 阻塞等待结果
```

适用场景：需要获取线程执行结果时（如异步计算、并行查询）。

**三种方式对比**

| 特性 | Thread | Runnable | Callable+FutureTask |
| --- | --- | --- | --- |
| 继承限制 | 有（单继承） | 无 | 无 |
| 返回值 | 无 | 无 | 有 |
| 异常 | 不可抛受检异常 | 不可抛受检异常 | 可抛受检异常 |
| 使用场景 | 简单演示 | 最常用 | 需要返回值 |

## 线程控制方法

| 方法 | 作用 | 要点 |
| --- | --- | --- |
| `sleep(ms)` | 暂停当前线程指定毫秒数 | 进入 TIMED_WAITING，不释放锁，到时间后进入就绪态等待调度 |
| `interrupt()` | 中断线程 | 目标线程处于 sleep/wait/join 阻塞态时抛 `InterruptedException` 并清除中断标志；运行态时仅设置标志位，需自行检查 `isInterrupted()`；不会自动传递给子线程 |
| `join()` | 等待目标线程执行完毕后再继续 | 底层通过 `wait()` 实现，常用于等待子线程结果 |
| `setDaemon(true)` | 设为守护线程 | 须在 `start()` 前调用；所有非守护线程结束后 JVM 退出，守护线程随之终止（典型如 GC 线程）；不能持有需要主动关闭的资源 |
| `yield()` | 提示调度器让出 CPU | 调度器可忽略该提示；让出后进入就绪态，可能立即被重新调度 |

## 线程生命周期

线程有 6 种状态（`Thread.State` 枚举）：

| 状态 | 含义 |
| --- | --- |
| NEW | 线程已创建，未调用 start() |
| RUNNABLE | 已调用 start()，等待或正在执行（含就绪和运行） |
| BLOCKED | 等待获取 synchronized 锁 |
| WAITING | 调用了 wait()/join()/LockSupport.park()，无限期等待 |
| TIMED_WAITING | 调用了 sleep(ms)/wait(ms)/join(ms) 等，有时限等待 |
| TERMINATED | run() 执行完毕或异常退出 |

状态转换核心路径：NEW 经 `start()` 进入 RUNNABLE；RUNNABLE 因竞争 synchronized 锁失败转 BLOCKED，获取到锁后转回 RUNNABLE；调用 `wait()`/`join()` 转 WAITING，被唤醒或等待对象结束后转回 RUNNABLE；调用 `sleep(ms)`/`wait(ms)` 转 TIMED_WAITING，超时或被唤醒后转回 RUNNABLE；`run()` 执行结束进入 TERMINATED。

## 线程安全

多个线程同时访问共享的可变数据时可能出现以下问题：

- **竞态条件（Race Condition）**：结果依赖线程执行时序，例如 `count++` 不是原子操作（读→改→写），多线程下结果不可预测
- **数据竞争（Data Race）**：多个线程同时访问同一内存位置，且至少一个是写操作，没有任何同步机制
- **可见性问题**：一个线程修改了变量值，另一个线程可能看不到最新值（因 CPU 缓存/指令重排）

**三大特性**

| 特性 | 含义 | 保证手段 |
| --- | --- | --- |
| 原子性 | 一个或多个操作要么全部执行成功要么全部不执行，中间不可被中断 | synchronized、Atomic 类 |
| 可见性 | 一个线程对共享变量的修改，其他线程能立即看到 | volatile（内存屏障）；synchronized 释放锁时将工作内存数据刷回主内存 |
| 有序性 | 程序按代码先后顺序执行，不受指令重排干扰 | volatile、synchronized 均可禁止特定的重排序 |

## 线程同步

**synchronized**：Java 内置关键字，给对象加互斥锁，同一时刻只允许一个线程进入同步块/方法，Java 多线程的锁都是基于对象的；是可重入锁，同一线程已持有锁时可再次获取而不会死锁。

```Java
// 同步实例方法：锁的是当前实例对象（this）
public synchronized void method() { }
// 同步静态方法：锁的是当前类的 Class 对象
public static synchronized void staticMethod() { }
// 同步代码块：锁的是括号中指定的对象
synchronized (lockObj) { }
```

锁升级（JDK 6 优化）：无锁 → 偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁，只能升级不能降级，锁状态记录在对象头 Mark Word 中。JDK 15 起偏向锁默认关闭，JDK 18 正式移除（撤销操作需 STW，得不偿失）。

| 锁级别 | 适用场景 | 实现方式 |
| --- | --- | --- |
| 偏向锁 | 只有一个线程访问 | 在对象头记录线程 ID |
| 轻量级锁 | 少量线程短暂竞争 | CAS 自旋 |
| 重量级锁 | 竞争激烈 | 操作系统互斥量，线程阻塞 |

底层通过字节码 `monitorenter`/`monitorexit` 指令实现：每个 Java 对象在 JVM 内部都关联一个 Monitor（监视器/锁）。

**volatile**：保证变量的可见性和有序性（禁止指令重排），但不保证原子性。适用于一写多读的标志位场景（如停止标志），以及双重检查锁定（DCL）单例模式中防止指令重排。

```Java
volatile boolean running = true;
```

**JUC 并发工具**：`java.util.concurrent`（JUC）是 Java 5 引入的并发工具包，大部分锁和同步器基于 AQS（AbstractQueuedSynchronizer）框架构建，AQS 通过一个 int 类型的 state 变量和一个 FIFO 等待队列管理线程的获锁与排队。

**ReentrantLock**：功能与 synchronized 类似，但支持公平锁、可中断锁、tryLock 超时、多条件变量（Condition），必须在 finally 中手动 `unlock()`。

```Java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();
}
```

**ReentrantReadWriteLock**：读写分离锁，读锁共享（多线程可同时读）、写锁独占，读写互斥、写写互斥，适用于读多写少场景。

**StampedLock**（JDK 8+）：支持乐观读的读写锁，基于乐观锁思想——不悲观加锁，而是先操作再通过 stamp 校验数据一致性，校验失败则升级为悲观读锁，性能优于 ReentrantReadWriteLock；不可重入，不支持 Condition。乐观读流程：`tryOptimisticRead()` 获取 stamp，读取数据后用 `validate(stamp)` 校验数据是否被写锁修改过，校验失败则升级为悲观读锁。

**Semaphore**：信号量，控制同时访问某资源的线程数，`acquire()` 获取许可、`release()` 释放许可，适用于限流、连接池等场景。

```Java
Semaphore semaphore = new Semaphore(3); // 最多3个线程同时访问
semaphore.acquire();
try {
    // 访问资源
} finally {
    semaphore.release();
}
```

**同步工具对比**

| 特性 | synchronized | ReentrantLock | ReentrantReadWriteLock | StampedLock |
| --- | --- | --- | --- | --- |
| 类型 | 内置关键字 | JUC 类 | JUC 类 | JUC 类 |
| 公平性 | 非公平 | 可选 | 可选 | 不支持 |
| 可中断 | 不支持 | 支持 | 支持 | 支持 |
| 超时获取 | 不支持 | 支持 | 支持 | 支持 |
| 条件变量 | 单条件(wait/notify) | 多条件(Condition) | 多条件 | 不支持 |
| 读写分离 | 不支持 | 不支持 | 支持 | 支持+乐观读 |
| 可重入 | 是 | 是 | 是 | 否 |

## 线程池

频繁创建和销毁线程开销大，线程池通过复用线程来降低开销、控制并发数量、统一管理线程生命周期。

`ThreadPoolExecutor` 是 Java 线程池的核心实现类，有 7 个核心参数：

```Java
public ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数（常驻）
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 非核心线程空闲存活时间
    TimeUnit unit,           // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler   // 拒绝策略
)
```

**任务提交流程：**

1. 当前线程数 < corePoolSize → 创建核心线程执行
2. 核心线程已满 → 任务放入 workQueue
3. 队列已满 + 线程数 < maximumPoolSize → 创建非核心线程执行
4. 队列已满 + 线程数 = maximumPoolSize → 执行拒绝策略

**拒绝策略：**

| 策略 | 行为 |
| --- | --- |
| AbortPolicy（默认） | 抛出 RejectedExecutionException |
| CallerRunsPolicy | 由提交任务的线程自己执行 |
| DiscardPolicy | 静默丢弃任务 |
| DiscardOldestPolicy | 丢弃队列中最早的任务，重新提交当前任务 |

不推荐使用 `Executors` 工厂方法创建线程池：

- `newFixedThreadPool`/`newSingleThreadExecutor` 使用无界队列 `LinkedBlockingQueue`，任务堆积可导致 OOM
- `newCachedThreadPool` 的 maximumPoolSize 为 `Integer.MAX_VALUE`，可能创建大量线程导致 OOM

推荐直接使用 `ThreadPoolExecutor` 构造器，明确指定每个参数。

**线程数配置：**

- CPU 密集型：线程数 = CPU 核心数 + 1（+1 是为了在某线程因缺页中断等短暂阻塞时保持 CPU 利用率）
- IO 密集型：线程数 = CPU 核心数 × 2（或更高，因 IO 等待时 CPU 空闲）
- 精确公式：线程数 = CPU 核心数 × (1 + 等待时间/计算时间)，生产环境通常仍需压测调优

**优雅关闭：**

| 方法 | 作用 |
| --- | --- |
| `shutdown()` | 不再接受新任务，等待已提交任务执行完毕 |
| `shutdownNow()` | 尝试中断所有线程，返回未执行的任务列表 |
| `awaitTermination(timeout, unit)` | 阻塞等待线程池终止，超时返回 false |

优雅关闭模式：先 `shutdown()`，再 `awaitTermination()` 等待，超时则 `shutdownNow()`。

## 常见并发问题

**死锁**：两个或多个线程互相持有对方需要的锁，导致所有线程永久阻塞，发生后没有任何机制能自动解除，只能强制结束 JVM 进程。

四个必要条件（Coffman 条件）：

| 条件 | 含义 |
| --- | --- |
| 互斥 | 资源同一时间只能被一个线程持有 |
| 占有并等待 | 持有一个资源，同时等待另一个 |
| 不可剥夺 | 资源只能由持有者主动释放 |
| 循环等待 | 线程间形成等待环路 |

```Java
// 死锁示例
static final Object lockA = new Object();
static final Object lockB = new Object();
// Thread-A: synchronized(lockA) { synchronized(lockB) { } }
// Thread-B: synchronized(lockB) { synchronized(lockA) { } }
// A持有lockA等lockB，B持有lockB等lockA → 死锁
```

解决方案：

1. **固定加锁顺序**（破坏循环等待）：所有线程按相同全局顺序申请锁
2. **tryLock 超时放弃**（破坏占有并等待）
3. **一次性申请所有资源**（破坏占有并等待）

检测工具：`jstack`（`jps` 找到 PID 后执行 `jstack <PID>`，输出中会显示 "Found one Java-level deadlock" 及详细信息）或 Arthas 的 `thread -b` 命令（一键定位阻塞其他线程的线程）。

**ThreadLocal**：为每个线程提供变量的独立副本，实现线程隔离，避免共享资源竞争，本质是"空间换时间"。核心方法：`set(value)` 设置值、`get()` 获取值、`remove()` 移除值。典型场景：存储用户会话信息、数据库连接、Spring 事务上下文、traceId 等。

```Java
static final ThreadLocal<String> userIdHolder = new ThreadLocal<>();
// 使用
userIdHolder.set("user-001");
String userId = userIdHolder.get();
// 必须在 finally 中清理
userIdHolder.remove();
```

与 synchronized 的区别：synchronized 是多线程共享同一变量并加锁互斥；ThreadLocal 是每个线程独立副本，无共享无需加锁。`ThreadLocalMap` 是 ThreadLocal 的静态内部类，是定制的哈希表，以 ThreadLocal 实例为 key、值为 Entry 存放。

内存泄漏原因：线程池中的线程会被复用、不会主动结束，导致 ThreadLocalMap 中的 Entry 不会被回收。必须在每次使用完后调用 `remove()` 清理。

## 应用场景

| 场景 | 首选方案 | 注意事项 |
| --- | --- | --- |
| Web 请求隔离 | ThreadLocal + 线程池 | 用完必须 remove() |
| 异步非阻塞操作 | CompletableFuture | 注意异常处理 |
| 大数据并行计算 | parallelStream / ForkJoin | 注意线程安全的聚合操作 |
| 等待多个子任务完成 | CountDownLatch | 计数归零前主线程一直阻塞 |
| 生产者-消费者 | BlockingQueue | 选有界队列防 OOM |
| 本地缓存初始化 | DCL + ReentrantLock | volatile 不可少 |
| 缓存并发读写 | ConcurrentHashMap | 线程安全容器，无需自行加锁 |
| 并发资源限制 | Semaphore | 务必 finally 释放 |
| 多任务结果编排 | CompletableFuture.allOf | 设置合理超时 |
