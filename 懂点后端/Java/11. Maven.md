# Maven

## Maven 是什么

Maven 是 Apache 基金会旗下的开源项目管理和构建工具，主要用于 Java 项目，基于 **POM（Project Object Model，项目对象模型）** 的概念，通过一段 XML 配置管理项目的构建、依赖和文档。用前端的视角类比：**Maven 之于 Java，就像 npm/pnpm 之于 js**，两者解决的核心问题是一样的——依赖管理和项目构建。官网：[https://maven.apache.org/](https://maven.apache.org/)

Maven 的三大核心作用：

- **依赖管理**（类比 `npm install`）：管理项目依赖的 jar 包，避免版本冲突。没有 Maven 之前需要手动从官网下载 jar 包导入项目，就像前端在没有 npm 之前手动下载 jQuery 的 js 文件；有了 Maven 之后只需在 `pom.xml` 中声明一段配置，Maven 就会自动下载依赖，这和在 `package.json` 中声明依赖后执行 `npm install` 是一个道理。
- **项目构建**（类比 `npm run build`）：提供标准化、跨平台的自动化构建方式，通过命令完成编译（compile）、测试（test）、打包（package）、发布（deploy），理念上和前端的 `npm run build`、`npm run test` 等脚本命令完全一致。
- **统一项目结构**（类比脚手架生成的目录结构）：不同 Java IDE（IDEA、Eclipse）创建的项目目录结构可能不同，Maven 提供一套标准目录结构，使项目能在不同 IDE 间无缝切换，就像前端用 Vue CLI 或 Create React App 创建的项目都有统一目录规范一样。

```xml
<!-- Maven 的依赖声明，类似 package.json 中的 dependencies -->
<dependencies>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context</artifactId>
        <version>6.1.4</version>
    </dependency>
</dependencies>
```

## 核心概念

**POM 文件**（类比 `package.json`）：`pom.xml` 是 Maven 项目的核心配置文件，相当于前端项目中的 `package.json`，描述项目的基本信息、依赖关系、构建配置。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0">  <!-- pom文件的根标签，表示当前maven项目 -->
    <modelVersion>4.0.0</modelVersion>  <!-- 声明项目描述遵循哪一个POM模型版本 -->

    <!-- 项目坐标，类似 package.json 中的 name + version -->
    <groupId>com.example</groupId>        <!-- 组织名，类似 npm 的 scope -->
    <artifactId>maven-project01</artifactId> <!-- 模块名，类似 package name -->
    <version>1.0-SNAPSHOT</version>        <!-- 版本号 -->

    <!-- 编译配置 -->
    <properties>
        <maven.compiler.source>17</maven.compiler.source>  <!-- 编译JDK的版本 -->
        <maven.compiler.target>17</maven.compiler.target>  <!-- 运行JDK的版本 -->
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding> <!-- 设置项目的字符集 -->
    </properties>
</project>
```

| Maven (pom.xml) | npm (package.json) | 说明 |
|---|---|---|
| groupId | scope (@vue) | 组织/团队标识 |
| artifactId | name | 项目/模块名称 |
| version | version | 版本号 |
| `<dependencies>` | dependencies | 项目依赖 |
| `<build><plugins>` | scripts + devDependencies | 构建工具和插件 |

**Maven 坐标**（类比 npm 包名 + 版本号）是资源的唯一标识，由三部分组成：**groupId**（组织名称，通常是域名反写，如 `com.example`，类似 npm 的 scope）、**artifactId**（模块名称，如 order-service，类似 npm 的包名）、**version**（版本号，`SNAPSHOT` 为开发中快照版本，`RELEASE` 为稳定发行版本）。三者组合能在全球范围内唯一定位一个 Java 库，就像 `@vue/cli@5.0.8` 能唯一定位一个 npm 包一样。

**Maven 仓库**（类比 npm registry）分三种：

- **本地仓库**：本机缓存已下载 jar 包的目录，类比前端的 `node_modules` 或 pnpm 的全局 store。
- **中央仓库**：Maven 官方维护的全球仓库（[https://repo1.maven.org/maven2/](https://repo1.maven.org/maven2/)），类比 npm 官方 registry。
- **远程仓库（私服）**：公司内部搭建的私有仓库，类比公司内部的 npm 私有源（如 Verdaccio、Nexus）。

依赖查找顺序：本地仓库 → 远程仓库（私服） → 中央仓库，和 npm 配置了私有源后的查找逻辑一样。搜索 Maven 依赖的网站 [https://mvnrepository.com/](https://mvnrepository.com/) 类似前端的 [npmjs.com](https://www.npmjs.com/)。

Maven 项目目录结构（类比前端项目结构）：

```
maven-project/
├── src/
│   ├── main/                    # 源代码目录（类比 src/）
│   │   ├── java/                # Java 源代码（类比 src/components/）
│   │   └── resources/           # 配置文件（类比 public/ 或 src/assets/）
│   └── test/                    # 测试代码目录（类比 tests/ 或 __tests__/）
│       ├── java/                # 测试代码
│       └── resources/           # 测试配置文件
├── target/                      # 编译打包输出目录（类比 dist/）
└── pom.xml                      # 项目配置文件（类比 package.json）
```

## 依赖管理

在 `pom.xml` 中通过 `<dependencies>` 标签声明依赖：

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context</artifactId>
        <version>6.1.4</version>
    </dependency>
</dependencies>
```

添加依赖后需要刷新，类似前端修改 `package.json` 后重新 `npm install`；本地仓库没有对应 jar 包时，Maven 会自动从远程仓库下载。

**依赖传递**（类比 npm 的依赖树）：A 依赖 B、B 依赖 C，则 A 会自动拥有 C 的依赖，和 npm 安装一个包时自动安装其所有子依赖是一个道理。

**排除依赖**（类比 npm 的 overrides/resolutions）：传递下来的依赖不需要时，可通过 `<exclusions>` 主动排除，类似前端用 `overrides`（npm）或 `resolutions`（yarn）控制依赖树。

```xml
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-context</artifactId>
    <version>6.1.4</version>
    <exclusions>
        <exclusion>
            <groupId>io.micrometer</groupId>
            <artifactId>micrometer-observation</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

**依赖范围 scope**（类比 dependencies vs devDependencies）：

| scope 值 | 主代码 | 测试代码 | 打包 | 前端类比 |
|---|---|---|---|---|
| compile（默认） | ✅ | ✅ | ✅ | dependencies |
| test | ❌ | ✅ | ❌ | devDependencies（仅测试用） |
| provided | ✅ | ✅ | ❌ | peerDependencies |
| runtime | ❌ | ✅ | ✅ | 运行时依赖 |

**provided**：编译和测试时需要，打包部署时不需要包含，因为运行环境会提供。典型例子是 Servlet API——开发 Web 应用时代码用到 `HttpServletRequest` 等类，编译需要，但部署到 Tomcat 后 Tomcat 自身已包含 Servlet API，若 jar 包里再打包一份反而冲突。前端类比：这就像 `peerDependencies`，比如开发 Vue 组件库时代码里 `import Vue from 'vue'`，但打包不会把 Vue 打进去，因为使用该组件库的项目一定已安装 Vue。

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>  <!-- Tomcat 会提供，不需要打包 -->
</dependency>
```

**runtime**：编译时不需要（代码不会直接 import 它的类），运行时才需要。典型例子是 JDBC 驱动——代码只使用 JDK 自带的 `java.sql.Connection`、`java.sql.DriverManager` 等标准接口，不直接引用 MySQL 驱动的类，但运行时 JVM 需要通过驱动实际连接数据库。前端类比：有点像动态 `import()` 加载的模块——编译时不需要静态分析，运行时才加载执行；也类似 polyfill，业务代码不会直接引用 `core-js`，但运行时需要它提供兼容性支持。

```xml
<dependency>
    <groupId>mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.33</version>
    <scope>runtime</scope>  <!-- 编译时不需要，运行时加载 -->
</dependency>
```

**scope 对依赖传递的影响**：`test` 和 `provided` 范围的依赖不会被传递。例如项目 A 依赖项目 B，B 中有一个 `scope=test` 的 JUnit 依赖，A 引入 B 时 JUnit 不会被传递到 A，这和前端的 `devDependencies` 类似——`npm install` 一个包时，该包的 `devDependencies` 不会被安装到自己项目中。

## 生命周期

Maven 将项目构建过程抽象为三套相互独立的生命周期（类比 npm scripts 的执行流程）：

| 生命周期 | 作用 | 前端类比 |
|---|---|---|
| clean | 清理工作，删除上一次构建生成的文件 | `rm -rf dist/` |
| default | 核心构建流程，包含编译、测试、打包、安装、部署等阶段 | — |
| site | 生成项目文档和报告 | — |

每套生命周期包含若干有顺序的阶段（phase），后面的阶段依赖前面的阶段。常用阶段：`clean`（移除上一次构建生成的文件）、`compile`（编译项目源代码）、`test`（用 JUnit 等框架运行测试）、`package`（打包为 jar/war 等）、`install`（安装项目到本地仓库）。同一套生命周期中，执行后面的阶段时前面的阶段都会先执行。
