# Spring IOC

## IOC 与 DI

Spring IOC（Inversion of Control，控制反转）解决的是对象之间的依赖管理：对象由谁创建、依赖由谁传入、初始化由谁触发、销毁由谁收尾。传统写法中业务类直接 `new` 依赖对象，类与类互相绑死，难以替换实现和测试；IOC 把依赖关系显式交给容器管理，业务类只需声明自己需要什么，具体依赖由容器送到对象手里。

| 维度 | 传统写法 | Spring 写法 |
|------|---------|------------|
| 对象创建 | 业务类自己 `new` | 容器统一创建 |
| 依赖关系 | 写死在类内部 | 由配置或注解决定 |
| 替换实现 | 修改业务代码 | 替换 Bean 即可 |
| 测试成本 | 依赖难替换 | 依赖容易注入 |

```Java
// 面向接口编程，OrderService 不再关心具体通知方式
public interface NoticeClient { void send(String msg); }

public class OrderService {
    private final NoticeClient noticeClient;
    public OrderService(NoticeClient noticeClient) {
        this.noticeClient = noticeClient;
    }
}
```

DI（Dependency Injection）是 IOC 最常见的实现方式：IOC 是思想，DI 是动作，容器创建对象后把依赖传进去。

| 注入方式 | 写法 | 推荐程度 | 适用场景 |
|---------|------|---------|---------|
| 构造方法注入 | 构造方法传参 | 推荐 | 必需依赖 |
| Setter 注入 | set 方法传参 | 可用 | 可选依赖 |
| 字段注入 | 字段上写 `@Autowired` | 可用 | 可选依赖 |

优先使用构造方法注入：依赖关系在对象创建时就完整建立，方便单元测试，且容器缺少所需 Bean 时会在启动阶段直接报错，而不是留到线上调用时出空指针。

## 容器与 Bean

`ApplicationContext` 是 Spring 最常用的 IOC 容器接口，可理解为"对象工厂 + 对象仓库 + 生命周期管理器"，负责读取配置、扫描组件、创建 Bean、解析依赖，同时处理初始化回调、事件发布、国际化等能力。

容器启动流程：

```
配置加载 → 组件扫描 → 生成 BeanDefinition → 实例化对象 → 依赖注入 → 初始化回调 → Bean 就绪
```

"找不到 Bean"通常出现在依赖注入阶段，"循环依赖"出现在实例化阶段。

```Java
@Configuration
public class AppConfig {
    @Bean
    public NoticeClient noticeClient() { return new SmsNoticeClient(); }
    @Bean
    public OrderService orderService(NoticeClient client) {
        return new OrderService(client);
    }
}
// 启动容器
ApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
context.getBean(OrderService.class).create("O1001");
```

被容器管理的对象叫 Bean，本质仍是 Java 对象，只是创建、依赖、初始化、销毁由 Spring 接管。

| 概念 | 含义 |
|------|------|
| Bean Class | 生成 Bean 的 Java 类 |
| Bean Name | 容器中的 Bean 名称 |
| BeanDefinition | Bean 的说明书 |
| Bean Instance | 运行期真正创建的对象 |

命名规则：`@Bean` 未指定名称时以方法名为 Bean Name；`@Component` 扫描的类默认以类名首字母小写为 Bean Name。

什么不该放进容器：

- 纯数据对象（DTO/VO/PO）
- 频繁创建的临时对象（Builder、迭代器）
- 有状态且线程不安全的对象

判断标准：需要被多个组件共享、需要被注入依赖、或需要容器管理生命周期的对象适合做 Bean，否则直接 `new`。

## 配置方式

**注解配置**：在类上使用注解，Spring 通过组件扫描注册为 Bean。

| 注解 | 常见位置 | 语义 |
|------|---------|------|
| `@Component` | 通用组件 | Spring 组件 |
| `@Service` | 业务服务类 | 业务层对象 |
| `@Repository` | DAO/Mapper | 数据访问对象 |
| `@Controller` | Web 层 | MVC 控制器 |

`@ComponentScan` 扫描指定包及子包，`@SpringBootApplication` 内部已包含组件扫描能力。

**Java Config**：使用 `@Configuration` + `@Bean` 编写配置类，适合配置第三方对象（无法修改源码加 `@Component`），或需要明确控制创建过程的对象。

```Java
@Configuration
public class PayConfig {
    @Bean
    public PaymentClient paymentClient() {
        return new PaymentClient("test-app-key");
    }
}
```

业务类优先用注解扫描，第三方对象优先用 Java Config，两者可混用；Spring 还支持 XML 配置（`<bean>` 标签），老项目或外部化配置场景下仍有用，新项目已较少使用。

## 依赖注入写法

**构造方法注入（推荐）**：适合必需依赖，字段可声明为 `final`，脱离 Spring 也容易测试（直接传入 mock）。Spring 4.3 以后，只有一个构造方法时 `@Autowired` 可省略。

```Java
@Service
public class UserService {
    private final UserRepository userRepository;
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

**Setter 注入（可选依赖）**：适合非必需的增强性依赖，Bean 没有该依赖也能正常工作。

```Java
@Autowired(required = false)
public void setAuditService(AuditService auditService) {
    this.auditService = auditService;
}
```

**字段注入（不推荐用于新代码）**：写法最省事但缺陷明显：依赖关系不显式（看构造方法看不出依赖了谁）、字段不能声明为 `final`、单元测试需借助反射或 Spring 测试框架注入 mock。已有代码可逐步重构为构造方法注入。

**`@Autowired` 查找规则**：

1. 按类型匹配 — 同类型只有一个 Bean，直接注入
2. 多个同类型 Bean — 结合参数名/字段名匹配 Bean Name
3. 仍无法确定 — 使用 `@Qualifier` 指定，或标记 `@Primary`
4. 以上都失败 — 启动报错 `NoUniqueBeanDefinitionException`

**多实现注入**：`@Qualifier` 按名称精确指定要注入的 Bean；`@Primary` 标记默认实现，类型冲突时优先被选择，适合"大多数场景用 A，个别场景用 B"；也可将所有同类型 Bean 自动注入到 `Map<String, T>`（key 为 Bean Name），适用于策略模式、插件化场景。

```Java
@Service
public class NoticeDispatcher {
    private final Map<String, NoticeClient> clientMap;
    public NoticeDispatcher(Map<String, NoticeClient> clientMap) {
        this.clientMap = clientMap;
    }
    public void send(String type, String msg) {
        NoticeClient client = clientMap.get(type + "NoticeClient");
        if (client != null) client.send(msg);
    }
}
```

## 生命周期与作用域

生命周期：

```
实例化对象 → 填充依赖（DI）→ 初始化回调 → 放入容器使用 → 容器关闭时销毁回调
```

初始化回调常用 `@PostConstruct`（最常用）、`InitializingBean`、`@Bean(initMethod=...)`，适合加载轻量缓存、校验配置、预热本地资源，不宜放耗时的远程调用，否则会拖慢启动。

```Java
@Service
public class CityService {
    private final Map<String, String> cityMap = new HashMap<>();
    @PostConstruct
    public void init() {
        cityMap.put("010", "北京");
        cityMap.put("021", "上海");
    }
}
```

销毁回调在容器关闭时释放资源（关闭线程池、断开连接等），常用 `@PreDestroy`、`DisposableBean`、`@Bean(destroyMethod=...)`；只有被容器管理的 Bean 才会触发，自己 `new` 的对象不会。

作用域：

| 作用域 | 含义 | 常见场景 |
|--------|------|---------|
| `singleton` | 容器中只有一个实例 | 默认，大多数 Service/Repository |
| `prototype` | 每次获取都创建新实例 | 短生命周期有状态对象 |
| `request` | 每个 HTTP 请求一个实例 | Web 请求上下文 |
| `session` | 每个 Session 一个实例 | 用户会话状态（极少直接使用） |

常见陷阱：Singleton 注入 Prototype 时，Prototype 实例只在 Singleton 创建时注入一次，之后不会再创建新的。解决方式：`@Lookup` 注解、注入 `ObjectFactory<T>` 或 `Provider<T>`、或通过 `ApplicationContext.getBean()` 每次获取新实例。

```Java
@Service
public class TaskService {
    private final ObjectFactory<TraceIdHolder> traceIdFactory;
    public TaskService(ObjectFactory<TraceIdHolder> traceIdFactory) {
        this.traceIdFactory = traceIdFactory;
    }
    public void execute() {
        String traceId = traceIdFactory.getObject().getTraceId();
    }
}
```

## 注入注解对比

- **`@Autowired`（Spring 原生）**：默认按类型匹配，可用在字段、Setter、构造方法上。
- **`@Resource`（Java 标准 JSR-250）**：来自 `javax.annotation` 包，不依赖 Spring；默认按名称匹配（先按字段名匹配 Bean Name，找不到再按类型），可通过 `name` 属性精确指定；只能用在字段和 Setter 上，不能用在构造方法参数上。查找顺序对比：`@Autowired` 类型→名称→`@Qualifier`/`@Primary`；`@Resource` 名称→类型。
- **`@Inject`（Java 标准 JSR-330）**：来自 `javax.inject` 包，行为与 `@Autowired` 基本一致，默认按类型匹配，没有 `required` 属性（依赖不存在直接报错），需额外引入依赖，适合需要脱离 Spring 框架（如 Google Guice）的场景。
- **`@RequiredArgsConstructor`（Lombok）**：为所有 `final` 字段生成构造方法，保留构造方法注入的优点（依赖显式、`final` 不可变、易测试），不需要手动维护构造方法；配合 `@Qualifier` 时注解写在 `final` 字段上即可。
- **`@Bean` 方法参数注入**：参数自动从容器按类型查找并注入，无需额外注解，常用于注册第三方类；同类型多个 Bean 时在参数上加 `@Qualifier` 指定。
- **`@Value` 属性注入**：注入配置属性值（来自 `application.yml`/`application.properties` 或环境变量），支持 SpEL 表达式和默认值语法 `${key:default}`，也可用在构造方法参数上以保持字段 `final`。

| 维度 | @Autowired | @Resource | @RequiredArgsConstructor（推荐） |
|------|-----------|-----------|------|
| 来源 | Spring | Java 标准 JSR-250 | Lombok |
| 默认匹配 | 按类型 | 按名称 | 按类型（构造方法注入） |
| 多实现处理 | 配合 `@Qualifier` | 通过 `name` 属性 | 配合 `@Qualifier` 写在字段上 |
| 可选标记 | `required = false` | 无直接支持 | 不支持（参数都是必需的） |
| 可用位置 | 字段、Setter、构造方法 | 字段、Setter | 类级别 |
| Spring 耦合 | 耦合 | 不耦合 | 不耦合（耦合 Lombok） |

选择口诀：必需依赖用构造方法（`@RequiredArgsConstructor`），可选依赖用 Setter（`@Autowired(required=false)`），按名称用 `@Resource` 或 `@Qualifier`，配置值用 `@Value`，第三方类用 `@Bean` 方法参数。

## 循环依赖

A 依赖 B，B 又依赖 A。Spring 对单例 Bean 的 setter/字段循环依赖有处理机制，但对构造方法循环依赖无法解决（创建 A 需要 B，创建 B 又需要 A，两个对象都无法先完成实例化）。循环依赖通常意味着模块边界不清晰，应优先从设计上拆解，而不是急着改成字段注入：重新审视职责（两个服务是否做得太多）、抽出第三个协调者、或将某个调用改成事件通知。

```Java
// 抽出协调者，让 OrderService 和 StockService 保持单一职责
@Service
public class OrderFlowService {
    private final OrderService orderService;
    private final StockService stockService;
    public OrderFlowService(OrderService orderService, StockService stockService) {
        this.orderService = orderService;
        this.stockService = stockService;
    }
    public void submit(String orderNo) {
        orderService.create(orderNo);
        stockService.lock(orderNo);
    }
}
```

`NoSuchBeanDefinitionException` 表示容器里没有符合条件的 Bean，常见原因：类没加组件注解、包没被扫描到、配置类没导入、条件注解不满足。Spring Boot 项目中启动类通常放在根包下，避免漏扫子包。
