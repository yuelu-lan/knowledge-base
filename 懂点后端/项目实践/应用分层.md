# 应用分层

分层的核心目的是**关注点分离**：每一层只做自己该做的事，层间通过接口解耦，使业务逻辑、外部依赖、协议适配各自独立演进。

## 各层职责

| 层 | 模块名 | 职责 |
| --- | --- | --- |
| **接入层** | Starter | 项目启动配置、服务配置、环境配置文件；接口协议适配、请求转发、响应封装、异常处理、日志打点、定时任务、消息监听 |
| **应用层** | Application | 用例实现与基础设施层接口定义；串联领域服务或基础设施服务，完成流程编排、聚合查询等工作流操作 |
| **领域层** | Domain | 业务逻辑实现；抽象领域对象，表达业务概念，承载业务行为和规则 |
| **基础设施层** | Infrastructure | 外部调用（数据库 CRUD、搜索引擎、文件系统、RPC 等）；领域防腐层（gateway 转译后才能在上层使用，依赖倒置解耦外部调用） |

## 核心概念

### Application Service vs Domain Service

|            | Application Service（应用服务） | Domain Service（领域服务） |
| ---------- | ------------------------------- | -------------------------- |
| **所在层** | 应用层                          | 领域层                     |
| **命名**   | `XxxApplication`                | `XxxDomainService`         |
| **职责**   | 对外暴露业务用例，负责编排      | 实现具体业务逻辑           |

两者都叫 `Service` 是因为本质上都是"服务"，`Application` 前缀用于区分层次。

### Repository

领域层定义接口（`IXxxRepository`），基础设施层提供具体实现。  
上层依赖接口而非实现，是依赖倒置的典型应用；Mapper 是 Repository 实现的底层组件。

### Mapper vs Manager

|  | Mapper | Manager |
| --- | --- | --- |
| **所在层** | 基础设施层 | 基础设施层 |
| **职责** | 直接操作 DB（MyBatis 接口），对应 PO 的 CRUD | 通用业务能力的封装与复用，位于 Mapper 之上 |
| **典型场景** | `selectById`、`insertBatch` 等单表操作 | 多 Mapper 组合、缓存处理、第三方服务封装 |
| **对应 DDD 概念** | Repository 的底层实现 | Gateway 的实现，或领域通用能力下沉 |

Mapper 只关心"怎么读写数据"，Manager 关心"怎么把数据和能力组合起来给上层用"。

> 注：Mapper 在某些上下文中也指对象转换工具（如 MapStruct），用于不同层对象之间的属性映射，与 MyBatis Mapper 是两个概念，需根据上下文区分。

## 三层架构 vs 四层架构

```
       三层架构                        四层架构

  ┌──────────────────┐          ┌──────────────────┐
  │     Starter      │          │     Starter      │
  └──────────────────┘          └──────────────────┘
          │ 依赖                         │ 依赖
          ▼                             ▼
  ┌──────────────────┐          ┌──────────────────┐
  │   Application    │          │   Application    │
  └──────────────────┘          └──────────────────┘
      ▲          △                      │ 依赖
  依赖(倒置)     实现                     ▼
      │          │              ┌──────────────────┐
  ┌──────────────────┐          │      Domain      │
  │  Infrastructure  │          └──────────────────┘
  └──────────────────┘              ▲          △
                                依赖(倒置)     实现
                                     │          │
                                ┌──────────────────┐
                                │  Infrastructure  │
                                └──────────────────┘
```

**如何选择：**

- **三层**：业务逻辑相对简单，领域概念不复杂，团队规模小；基础设施层接口定义放在 Application 层，结构更紧凑。
- **四层**：业务复杂度高，需要严格保护领域逻辑不被基础设施污染；独立的 Domain 层使核心业务规则可单独测试，边界更清晰。

### 命名规范参考

#### 类命名规范

| 层级           | 类型         | 命名规范            | 示例                  |
| -------------- | ------------ | ------------------- | --------------------- |
| Controller     | 控制器       | XxxController       | StudentController     |
| Application    | 应用服务     | XxxApplication      | EnrollmentApplication |
| Domain Service | 领域服务     | XxxDomainService    | GradeDomainService    |
| Domain Model   | 领域模型     | 业务名词            | Student, Course       |
| Repository     | 仓储接口     | XxxRepo             | StudentRepo           |
| Gateway        | 网关接口     | XxxGateway          | NotificationGateway   |
| Converter      | 转换器       | XxxConverter        | StudentConverter      |
| BO             | 业务对象     | XxxBO               | StudentPageBO         |
| DTO            | 数据传输对象 | XxxDTO              | GradeDTO              |
| Request        | 请求对象     | XxxReq/XxxRequest   | EnrollCourseReq       |
| Response       | 响应对象     | XxxResp/XxxResponse | CourseListResp        |

#### 方法命名规范

| 操作类型 | 命名规范                         | 示例                |
| -------- | -------------------------------- | ------------------- |
| 查询单个 | get/find/query + 实体名          | getByStudentNo      |
| 查询列表 | list/query + 实体名              | queryStudentPage    |
| 新增     | add/create/save + 实体名         | addStudent          |
| 修改     | update/modify + 实体名           | updateGrade         |
| 删除     | delete/remove + 实体名           | deleteCourse        |
| 校验     | validate/verify/check + 业务含义 | checkCourseConflict |

## 领域模型

| 对象 | 全称 | 说明 |
| --- | --- | --- |
| **DTO** | Data Transfer Object | 数据传输对象，细分为 Request（入参）、Response（返回）、DTO（封装对象），用于 RPC 或 HTTP 定义请求/响应 |
| **BO** | Business Object | 业务对象，由 Service 层输出的封装业务逻辑的对象 |
| **DO** | Domain Object | 领域对象，从现实世界抽象出的有形或无形的业务实体（四层架构引入） |
| **PO** | Persistent Object | 持久化对象，用于 DB、Cache 存储 |

**对象关系链：**

```
外部              接入层             应用层 / 领域层         基础设施层
Request/Response ──→ BO ──────────────────────────────→ PO（DB）
     DTO         ←── BO     （四层：BO ↔ DO ↔ PO）     DTO（RPC/HTTP）
```

## 数据模型流转

### 三层架构

```
                      客户端、网关、外部服务等
  request、DTO ↓                              ↑ response、DTO
┌──────────────────────────────────────────────────────────────────────────┐
│ 接入层（Starter）                                                          │
│  ┌────────────────────────────────────────────────────────┐              │
│  │  数据转换：xxRequest、xxResponse、xxDTO <---> BO         │              │
│  └────────────────────────────────────────────────────────┘              │
└──────────────────────────────────────────────────────────────────────────┘
                                    BO ↓          依赖 ↓          ↑ BO
┌──────────────────────────────────────────────────────────────────────────┐
│ 应用层 (application)               BO             BO              BO      │
└──────────────────────────────────────────────────────────────────────────┘
                                    BO ↓       ↑ 依赖倒置         ↑ BO
┌──────────────────────────────────────────────────────────────────────────┐
│ 基础设施层 (infrastructure)                                               │
│  ┌─────────────────────────┐   ┌───────────────────────────┐             │
│  │ 数据转换：xxBO <---> xxPO│   │ 数据转换：xxBO <---> xxDTO │              │
│  └─────────────────────────┘   └───────────────────────────┘             │
│  领域内：PO  PO                  领域外：Request  Response  DTO            │
└──────────────────────────────────────────────────────────────────────────┘
```
