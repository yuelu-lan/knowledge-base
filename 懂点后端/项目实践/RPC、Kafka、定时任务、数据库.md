# RPC、Kafka、定时任务、数据库

## RPC

**基本概念：** 内部服务之间的通信统一使用 RPC 协议，不互相提供 HTTP 接口。对外（外部系统）则一般使用 HTTP 或其他方式（如文件传输）。

**服务发现与注册中心：** RPC 服务启动时，自动将接口、端口、IP 等信息注册到统一的注册中心；调用方启动时通过基础框架自动获取服务端地址，开发者无需手动管理。

## 定时任务

**适用场景：** 无用户触发、需在特定时间执行的后端逻辑，如：

- 到期提醒（某笔业务到期前通知用户）
- 异步结果轮询（调用外部系统后定时查询结果）
- 失败重试补偿（调用下游超时后隔段时间重试）

**开发方式（XXL-Job）：** 在方法上加 `@XxlJob` 注解，在 XXL-Job 控制台创建任务并配置 Cron：

```java
@Component
public class OrderExpireNotifyJob {

    @XxlJob("orderExpireNotifyJob")
    public void execute() {
        // 查询即将到期订单，发送通知
    }
}
```

**Cron 表达式示例（秒 分 时 日 月 周）：**

| 表达式           | 含义              |
| ---------------- | ----------------- |
| `0 0 9 * * ?`    | 每天 09:00:00     |
| `0 */5 * * * ?`  | 每 5 分钟执行一次 |
| `0 0 0 1 * ?`    | 每月 1 号零点     |
| `0 0 10 ? * MON` | 每周一 10:00      |

**"最终一致性"：** 后端不要求强一致性，而是保证最终一致性——任何节点异常/超时，通过定时任务驱动重试，直到流程走完。涉及资金时需特别注意重试是否会导致重复操作。

## MQ 消息队列（Kafka）

**核心角色：** 发送方 → 消息队列 → 消费者

**三大作用：**

| 作用 | 说明 | 典型场景 |
| --- | --- | --- |
| 流程异步化 | 将非核心逻辑从主链路剥离，延迟处理 | 下单后通知买家、通知商家 |
| 领域间解耦 | 核心逻辑只保留最关键部分，后续扩展由独立服务消费消息完成 | 下单链路 + 营销/通知等独立服务 |
| 流量削峰 | 上游瞬时高流量由 MQ 缓冲，消费方按自己节奏处理 | 秒杀、大促场景 |

**使用注意事项：**

- **发送可能失败：** 需打日志、做告警，支持人工介入或触发重发。
- **消费幂等（重中之重）：** 发送方超时重试可能产生重复消息，消费方必须用业务唯一标识判断是否已处理，重复消息直接跳过。

```java
// 消费幂等示例
public void consume(OrderMessage message) {
    if (processedRecordService.exists(message.getBizId())) {
        return; // 已处理，直接跳过
    }
    // 执行业务逻辑
    doProcess(message);
    processedRecordService.save(message.getBizId());
}
```

## 存储选型

| 存储 | 核心用途 | 特点 | 典型场景 |
| --- | --- | --- | --- |
| **MySQL** | 在线业务主存储 | 强一致性，支持事务，数据量大时有慢查询风险 | 订单、用户、账单等核心业务数据 |
| **ES (Elasticsearch)** | 多条件复杂查询加速 | 每个查询字段建专有索引，只存查询条件字段 + ID | 订单列表搜索、报表筛选 |
| **Redis** | 缓存 + 分布式锁 | 扛流量能力远高于 MySQL，数据存内存 | 热点数据缓存、防并发写的分布式锁 |
| **Hive** | 离线大数据分析 | 存储量级极大，查询慢，在线应用不直接读 | 数仓分析、BI 报表 |

**ES 与 MySQL 的关系：** MySQL 是数据主存储，ES 是查询加速层。数据写入 MySQL 后同步到 ES，查询时先走 ES 拿到 ID，再回 MySQL 取完整数据（或直接返回 ES 中存的字段）。

**选型依据：** 方案设计阶段需评估非功能性需求——预估 QPS 量级、3~5 年数据存储量级，据此决定存储选型。
